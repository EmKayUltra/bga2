  <name>Task 1: Rewrite SceneManager for multi-player turn tracking and real server state</name>
  <files>apps/client/src/lib/engine/SceneManager.ts</files>
  <action>
    Rewrite SceneManager to replace Phase 1 hardcoded behavior with real game state management.

    **SceneManagerState update:**
    ```typescript
    export interface SceneManagerState {
      sessionId: string | null;
      currentPlayerIndex: number;      // replaces playerId
      playerNames: string[];           // from server state
      playerScores: number[];          // from server state
      currentFsmState: string;
      lastMoveResult: MoveResult | null;
      validMoves: ValidMove[];
      gameState: GameState | null;     // full state from server for UI binding
      turnHandoffMode: 'open-board' | 'pass-and-play';
      selectedSource: string | null;   // currently selected factory/center zone ID
      selectedColor: string | null;    // currently selected tile color
    }
    ```

    **init(sessionId) changes:**
    - Remove the hardcoded stubGameState. Instead, call `getGameState(sessionId)` to fetch real state.
    - Store the full GameState on SceneManagerState.gameState.
    - Derive playerNames, playerScores, currentPlayerIndex from the fetched state.
    - Pass the real state to AzulScene for rendering.
    - Still create the FSM (keep the XState machine for local turn state tracking), but initialize it with real state.

    **New interaction flow (two-step source → target):**
    1. Player taps a tile in a factory or center → SceneManager identifies the source zone and tile color → calls scene.selectSource(zoneId, color) which highlights all tiles of that color in the source AND highlights valid destination pattern lines (from validMoves filtering)
    2. Player taps a valid destination (pattern line slot or floor line) → SceneManager builds the move: `{ playerId: "player-{currentPlayerIndex}", action: "pick-tiles", source: selectedSourceZone, data: { color: selectedColor, patternLineRow: targetRow } }` → submits to server
    3. On valid response: SceneManager calls scene.animateMove(sourceZoneId, color, targetZoneId) → updates local state from server response → shows turn banner → enables next player

    **Move submission:**
    - `submitPlayerMove()` builds a Move with `action: "pick-tiles"`, `source` = factory/center zone ID, `data.color` = tile color, `data.patternLineRow` = target row (1-5, or 0 for floor-line direct)
    - After successful move: update gameState from response, derive next player info, call scene.updateFromState(newState) to re-render, notify onStateChange

    **Turn tracking:**
    - After each move response, check `state.phase`:
      - 'factory-offer' → normal play, enable interaction for currentPlayerIndex
      - 'wall-tiling' → the server handles this automatically (onRoundEnd was auto-called). The response state will have phase back to 'factory-offer' or 'finished'.
      - 'finished' → notify page to show score summary
    - Show turn change via `onTurnChange` callback (page listens and shows banner)

    **New callbacks for page integration:**
    ```typescript
    public onTurnChange: ((playerIndex: number, playerName: string) => void) | null = null;
    public onGameFinished: ((state: GameState) => void) | null = null;
    ```

    **Remove Phase 1 code:**
    - Remove hardcoded `playerId: 'player-1'`
    - Remove stub GameState creation
    - Remove piece-click-then-piece-click interaction (replace with source-then-destination)
  </action>
