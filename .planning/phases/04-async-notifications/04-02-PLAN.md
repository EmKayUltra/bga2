<objective>
Build the server-side notification pipeline: NotificationService (email via Resend + push dispatch), DeadlineService (Hangfire recurring job for expired deadlines + delayed reminder scheduling), and wire notification triggers into GameService.ValidateAndApplyMove.

Purpose: This is the core async game loop — when a player moves, the next player gets notified; when a deadline approaches, a reminder fires; when a deadline expires, the turn is auto-skipped. Without this, async games have no enforcement and no notifications.

Output: NotificationService.cs, DeadlineService.cs, extended GameService.cs, Hangfire recurring job registered in Program.cs.
</objective>

**1. Create NotificationService.cs** in apps/server/Services/. Constructor-inject: GameDbContext, IResend (from Resend package), PushServiceClient (from Lib.Net.Http.WebPush), ILogger, IConfiguration (for VAPID keys).

Methods:
- `NotifyYourTurn(Guid sessionId, string userId, int turnVersion)` — the main "your turn" dispatch. Flow:
  1. Check NotificationLog for existing {sessionId, turnVersion, userId, "email"} — skip if already sent (idempotency).
  2. Load NotificationPreference for userId (default to all-enabled if no row exists).
  3. If emailEnabled: look up user email via raw SQL `SELECT email FROM "user" WHERE id = @userId` (Better Auth user table in same DB). Send email via IResend with subject "It's your turn!" and HTML body containing player name + game link. Log to NotificationLog with channel="email".
  4. If pushEnabled: load all PushSubscription rows for userId. For each subscription, send push message via PushServiceClient with VAPID auth. Payload: `{ "title": "Your Turn!", "body": "It's your turn in your Azul game", "url": "/game/{sessionId}" }`. Log to NotificationLog with channel="push". Catch and log push failures (non-fatal; expired subscriptions may return 410).
  5. Email "From" address: `"BGA2 <noreply@bga2.dev>"`. Use `onboarding@resend.dev` as the actual From when RESEND_APITOKEN is the test key (Resend routes test-mode emails to account owner only). If RESEND_APITOKEN is empty/missing, log a warning and skip email entirely (graceful degradation for dev without Resend).
  6. If VAPID keys are empty/missing, log a warning and skip push entirely (graceful degradation for dev without VAPID setup).

- `SendDeadlineReminder(Guid sessionId, string userId, int turnVersion)` — similar to NotifyYourTurn but with "Deadline approaching" subject/body. Also checks NotificationLog with a distinct channel suffix (e.g., "email-reminder" / "push-reminder") for idempotency.

**2. Create DeadlineService.cs** in apps/server/Services/. Constructor-inject: GameDbContext, ILogger, GameService (to advance turn on timeout).

Methods:
- `ProcessExpiredDeadlines()` — Hangfire recurring job. Decorated with `[DisableConcurrentExecution(timeoutInSeconds: 60)]`. Flow:
  1. Query `GameTables.Where(t => t.IsAsync && !t.IsPaused && t.TurnDeadline <= DateTime.UtcNow && t.Status == TableStatus.Playing)`.
  2. For each expired table:
     a. Increment ConsecutiveSkipsCurrentPlayer.
     b. Check if ConsecutiveSkipsCurrentPlayer >= SkipThreshold AND SkipThreshold > 0 — if so, forfeit: set table.Status = Finished, publish via AppSync. Continue to next table.
     c. If not forfeit: advance turn — load the GameSession via table.SessionId, parse state to find currentPlayerIndex, advance to next player (increment currentPlayerIndex mod playerCount), set new TurnDeadline based on TimerMode hours, reset ConsecutiveSkipsCurrentPlayer to 0 (it tracks the NEW current player's consecutive skips, which starts at 0). Save changes.
     d. Trigger NotifyYourTurn for the new current player via Hangfire fire-and-forget.

```csharp
// Step 5c: Async game notification + deadline management
// Look up the GameTable for this session to check if it's async
var gameTable = await _db.GameTables.FirstOrDefaultAsync(t => t.SessionId == sessionId);
if (gameTable is { IsAsync: true, Status: TableStatus.Playing })
{
    // Calculate new deadline based on timer mode
    var timerHours = gameTable.TimerMode switch { "fast" => 12, "normal" => 24, "slow" => 72, _ => 24 };
    gameTable.TurnDeadline = DateTime.UtcNow.AddHours(timerHours);

    // Reset consecutive skips for the new current player (they just became active)
    gameTable.ConsecutiveSkipsCurrentPlayer = 0;

    // Cancel any previously scheduled reminder for the OLD player's turn
    // (The reminder Hangfire job IDs are stored as a convention: "reminder-{sessionId}-{oldVersion}")
    var oldReminderJobId = $"reminder-{sessionId}-{session.Version - 1}";
    BackgroundJob.Delete(oldReminderJobId);

    await _db.SaveChangesAsync();  // Save updated deadline on GameTable

    // Determine next player's userId from the new game state
    var (nextPlayerId, _) = ExtractPlayerAndRound(newStateJson);
    var nextUserId = ExtractUserIdForPlayer(newStateJson, nextPlayerId);

    if (!string.IsNullOrEmpty(nextUserId))
    {
        // Fire-and-forget: "your turn" notification
        BackgroundJob.Enqueue<NotificationService>(
            svc => svc.NotifyYourTurn(sessionId, nextUserId, session.Version));

        // Schedule reminder as delayed job
        var prefs = await _db.NotificationPreferences.FindAsync(nextUserId);
        var reminderHours = prefs?.ReminderHoursBeforeDeadline ?? 4;
        if (reminderHours > 0 && reminderHours < timerHours)
        {
            var delay = TimeSpan.FromHours(timerHours - reminderHours);
            var reminderJobId = $"reminder-{sessionId}-{session.Version}";
            BackgroundJob.Schedule<NotificationService>(
                svc => svc.SendDeadlineReminder(sessionId, nextUserId, session.Version),
                delay);
            // Note: Hangfire returns its own job ID. We use a convention-based ID for cancellation.
            // For proper cancellation, store the actual Hangfire job ID. Since BackgroundJob.Schedule returns string,
            // we can use it. However, BackgroundJob.Delete works with the returned ID, not a custom ID.
            // REVISED approach: store reminder Hangfire job ID on the GameTable or use a simpler pattern.
        }
    }
}
```

**IMPORTANT revision for reminder cancellation:** BackgroundJob.Schedule returns a string job ID, and BackgroundJob.Delete takes that job ID. We cannot use convention-based names. Instead:
- Add a `string? PendingReminderJobId` property to GameTable (simple nullable string column). Plan 01 may not have this — if missing, add it to GameTable.cs and GameDbContext config.
- When scheduling a reminder: `gameTable.PendingReminderJobId = BackgroundJob.Schedule<NotificationService>(...)`.
- When cancelling: `if (gameTable.PendingReminderJobId != null) { BackgroundJob.Delete(gameTable.PendingReminderJobId); gameTable.PendingReminderJobId = null; }`.
- Save the GameTable after updating PendingReminderJobId.
