  <name>Task 1: Update HookExecutor — BuildHookContext with players, add OnRoundEnd method</name>
  <files>apps/server/Services/HookExecutor.cs</files>
  <behavior>
    - BuildHookContext populates ctx.players from state.players (parsed from gameStateJson) — each with id, name, index
    - OnRoundEnd(hooksSource, gameStateJson, currentPlayer, round) executes onRoundEnd(ctx) and returns updated state JSON
    - ExtractPlayerAndRound reads currentPlayerIndex (int) and derives currentPlayer from state.players[currentPlayerIndex].id
    - BuildHookContext uses the same Jint engine instance as the caller (not creating its own separate engine) so ctx is accessible after setup
  </behavior>
  <action>
    Update HookExecutor.cs:

    1. **Fix BuildHookContext** — Currently sets `players: []`. Change to parse `gameStateJson`, extract the `players` array, and build a JavaScript array of `{ id, name, index }` objects in the context setup script. The context setup JS should be:
    ```
    var ctx = {
      state: JSON.parse('...'),
      currentPlayer: state.players[state.currentPlayerIndex].id,
      round: state.round,
      players: state.players.map(function(p, i) { return { id: p.id, name: p.name, index: i }; })
    };
    ```
    BUT — BuildHookContext currently creates its OWN Jint Engine to parse the context, then returns the JsValue. This is wrong — the calling method (GetValidMoves, OnMove) creates a SEPARATE engine. The ctx returned from BuildHookContext is from a DIFFERENT engine scope.

    **Fix pattern:** Change BuildHookContext to return the JavaScript STRING (not JsValue). The calling method (GetValidMoves, OnMove) executes this string on ITS engine before calling the hook function. This way ctx lives in the same Jint scope as the hook functions.

    Actually, looking more carefully: BuildHookContext returns a JsValue, but GetValidMoves creates its own engine and never uses the BuildHookContext engine. The current code has a latent bug — it works in Phase 1 because hooks are stubs. Fix: Make BuildHookContext return the setup JS string, and have GetValidMoves/OnMove/OnRoundEnd execute that string on their own engines.

    2. **Fix ExtractPlayerAndRound** — Currently reads `currentPlayer` string field. Change to read `currentPlayerIndex` (int) and look up player ID from `players[currentPlayerIndex].id`. Keep backward-compatible fallback for old state format.

    3. **Add OnRoundEnd method** — Same pattern as OnMove: create engine, execute shims + hooks source + context setup, invoke `onRoundEnd(ctx)`, extract mutated ctx.state via `JSON.stringify(ctx.state)`.

    4. **Update GetValidMoves and OnMove** — Use the new BuildHookContext string pattern (execute context setup on the hook engine, not a separate engine).

    Verify: `docker compose -f apps/infra/docker-compose.yml exec server dotnet build` succeeds with 0 errors.
