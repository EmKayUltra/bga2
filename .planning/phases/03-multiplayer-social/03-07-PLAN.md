  <name>Task 1: Wire real user IDs through CreateGame and ExtractPlayerResults</name>
  <files>apps/server/Services/GameService.cs, apps/server/Services/LobbyService.cs</files>
  <action>
    **GameService.CreateGame — add optional userIds parameter:**

    Change the signature from:
    ```csharp
    public async Task<CreateGameResponse> CreateGame(string gameId, string[] playerNames)
    ```
    to:
    ```csharp
    public async Task<CreateGameResponse> CreateGame(string gameId, string[] playerNames, string[]? userIds = null)
    ```

    The `userIds` parameter is optional (null for hot-seat games created without auth). When provided, `userIds[i]` corresponds to `playerNames[i]`.

    **GameService.CreateGame — add userId field to player objects:**

    In the players array builder (the `Select` around line 130), add a `userId` field to each player object:
    ```csharp
    var players = actualPlayerNames.Select((name, i) => new
    {
        id = $"player-{i}",
        name,
        userId = userIds != null && i < userIds.Length ? userIds[i] : (string?)null,
        score = 0,
        data = new { ... }  // unchanged
    }).ToArray<object>();
    ```

    The `id` field stays as `"player-{i}"` — it is the game-scoped player identifier used throughout hooks.ts and the FSM. The new `userId` field is the real Better Auth user ID, used only for match result recording. When `userIds` is null (hot-seat mode), `userId` will be null in the JSON.

    **GameService.ExtractPlayerResults — prefer userId field over id:**

    In `ExtractPlayerResults` (around line 498), change the player ID extraction to prefer the `userId` field if present:
    ```csharp
    // Prefer userId (real Better Auth ID) over id (game-scoped "player-{i}")
    var playerId = p.TryGetProperty("userId", out var userIdProp) &&
                   userIdProp.ValueKind == JsonValueKind.String
        ? userIdProp.GetString() ?? $"player-{idx}"
        : p.TryGetProperty("id", out var idProp)
            ? idProp.GetString() ?? $"player-{idx}"
            : $"player-{idx}";
    ```

    This ensures:
    - Lobby games: userId is the real Better Auth ID → match history works
    - Hot-seat games: userId is null in JSON → falls back to "player-{i}" → no regression
    - Old game states (before this fix): no userId field → falls back to id → backward compatible

    **Update the NOTE comment** on ExtractPlayerResults (currently says "player.id in the state is 'player-0'...") to reflect that userId is now available for lobby-created games.

    **LobbyService.StartGame — pass user IDs to CreateGame:**

    In `StartGame` (around line 289-291), extract both display names AND user IDs from the sorted TablePlayers list, and pass both to CreateGame:
    ```csharp
    var playerNames = players.Select(p => p.DisplayName).ToArray();
    var userIds = players.Select(p => p.UserId).ToArray();
    var gameResponse = await _gameService.CreateGame(table.GameId, playerNames, userIds);
    ```

    This is the only call site that needs to change. The existing `CreateGame` calls from hot-seat mode (POST /games endpoint in GameEndpoints.cs) pass only playerNames and userIds defaults to null — backward compatible.
