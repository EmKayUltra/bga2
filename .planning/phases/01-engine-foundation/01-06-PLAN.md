<objective>
Build the C# API server with a move validation endpoint that executes TypeScript game hooks via Jint, persists state to PostgreSQL, and returns validMoves[] for client-side highlighting.

Purpose: Server-authoritative validation is a core architectural principle. The client never enforces rules — it sends moves to the server, which validates them against the game's hook functions and returns the result along with legal moves for the next turn. This plan proves that pipeline works end-to-end.
Output: Working ASP.NET Core Minimal API with move endpoint, Jint hook execution, and PostgreSQL persistence.
</objective>

```typescript
interface Move {
  playerId: string;

  action: string;

  valid: boolean;

}
```


       - Constructor: inject GameDbContext, HookExecutor
       - `ValidateAndApplyMove(Guid sessionId, MoveRequest move)`:
         1. Load GameSession from DB
         2. Deserialize State JSON into game state object
         3. Call HookExecutor.GetValidMoves(state) to get legal moves
         4. Check if submitted move is in validMoves list
         5. If valid: call HookExecutor.OnMove(state, move) to mutate state
         6. Save updated state to DB with version increment
         7. Call HookExecutor.GetValidMoves(newState) for next player

  <action>
    Create `apps/server/Services/HookExecutor.cs`:

    1. **HookExecutor** class:
       - Constructor: accepts hooks JavaScript source code (compiled from TypeScript — for Phase 1, the hooks.ts stubs are simple enough to be valid JavaScript with type annotations stripped)
       - Uses Jint `Engine` to create a JavaScript execution environment
       - Loads the hook functions into the Jint engine

    2. **GetValidMoves(gameState: string, currentPlayer: string, round: int)**:
       - Create Jint Engine instance
       - Set the hook source code
       - Construct a HookContext object: { state: JSON.parse(gameState), currentPlayer, round, players: [] }
       - Call `getValidMoves(ctx)` in the Jint engine
       - Parse return value as ValidMove[] equivalent
       - Return list of valid moves

    3. **OnMove(gameState: string, move: MoveRequest)**:
       - Create Jint Engine instance
       - Load hook source and game state
       - Call `onMove(ctx, move)` in the Jint engine
       - Extract mutated state from context
       - Return updated state as JSON string

    4. **LoadHooks(gameId: string)**:
       - For Phase 1: load hooks from a known file path (libs/games/azul/src/hooks.ts — but stripped of TypeScript syntax to plain JS)
       - Create a simple TS-to-JS strip function that removes type annotations (`: type` patterns, `interface` blocks, `import type` statements)
       - This is a pragmatic Phase 1 approach. Phase 2 will use a proper TypeScript compiler step.

    5. **Error handling:**
       - Jint execution errors (syntax error, runtime error in hook) should be caught and returned as MoveResult.Errors
       - Jint timeout: set a maximum execution time (e.g., 5 seconds) to prevent infinite loops in hook code

    NOTE: Jint runs JavaScript, not TypeScript. For Phase 1, the hooks.ts stubs are simple enough that stripping type annotations produces valid JS. For Phase 2, a proper compile step (TypeScript -> JavaScript) will be added. Per research Open Question 2, Jint is the pragmatic Phase 1 choice — in-process, no Node.js sidecar needed.
