  <behavior>
    RED phase — write failing tests for:

    1. FSM initializes in the correct starting state (e.g., 'playing.draftingPhase')
    2. FSM transitions from drafting to tiling phase on PHASE_END event
    3. FSM transitions from tiling to scoring on appropriate event
    4. FSM transitions to 'finished' state on GAME_END event
    5. Invalid event in wrong state does NOT change state (e.g., GAME_END during drafting doesn't jump to finished — it should be ignored or only valid from specific states depending on design)
    6. Parallel regions: create a test machine with two parallel player regions. Send SUBMIT_P1 and SUBMIT_P2 events. onDone fires only after BOTH reach final state.
    7. Parallel regions: sending only SUBMIT_P1 does NOT trigger onDone
    8. FSM context holds GameState and round number; assign action updates context on transition
    9. FSM can be created with a dynamic config (createGameFSM takes a GameFSMConfig describing phases/stages)

    GREEN phase — implement:

    **types.ts:**
    - FSMContext: { gameState: GameState, round: number, currentPlayerIndex: number }
    - FSMEvent: union type of all game events (PHASE_END, GAME_END, SUBMIT_MOVE, ROUND_END, etc.)
    - FSMStateValue: type for nested state values
    - GameFSMConfig: { phases: PhaseConfig[], parallelTurns: boolean }
    - PhaseConfig: { id: string, stages: StageConfig[] }
    - StageConfig: { id: string }

    **GameFSM.ts:**
    - `createGameFSM(config: GameFSMConfig)` factory function that builds an XState machine from config
    - Default machine structure: Game (initial: 'playing') > playing (initial: first phase) > Phase (initial: 'turn') > Turn > Stage
    - Turn state supports `type: 'parallel'` when config.parallelTurns is true
    - Each parallel region has: waiting -> active -> done (final)
    - onDone on parallel turn state advances to next stage or triggers ROUND_END
    - Use `createMachine` from xstate v5 (NOT v4 `Machine` factory)
    - Use `createActor` to run the machine (NOT v4 `interpret`)
    - Context typed with FSMContext
    - assign actions to update gameState and round on transitions

    **index.ts:** barrel export createGameFSM, types, and config interfaces

    REFACTOR phase — clean up if needed, ensure all tests pass.
  </behavior>
  <implementation>
    Use XState v5 createMachine with nested states per research Pattern 2. The factory function allows different games to define their own phase/stage structure while the engine provides the runtime.

    Key XState v5 patterns:
    - `createMachine({ ... })` not `Machine({ ... })`
    - `createActor(machine)` not `interpret(machine)`
    - Parallel: `type: 'parallel'` on a state, with child states as regions
    - onDone on parallel: fires when ALL child regions reach a `type: 'final'` state
    - assign: `assign({ round: ({ context }) => context.round + 1 })`
    - Every parallel region MUST have at least one `type: 'final'` state for onDone to work
  </implementation>
</feature>
